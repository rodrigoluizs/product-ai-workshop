# Initiative document style

When working on an initiative document (`*-initiative.md`), enforce these
conventions. Initiatives are for leadership — they explain *why* and *what
outcome*, not *how*.

- **Outcome-first.** Lead with the user/business problem and the change we want
  to see, not the solution.
- **Required sections:** Problem · Goal / Desired Outcome · Why now · Success
  metric · Scope (and explicit Out of scope).
- **Executive-readable.** Plain language, no engineering jargon, no file names,
  no implementation detail.
- **One measurable success metric.** If it can't be measured, push back and ask
  for one.
- **Short.** Aim for one page. Cut anything that doesn't help a decision.
